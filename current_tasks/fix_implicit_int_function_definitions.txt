Fix: Support implicit-int return type for function definitions (C89/K&R)

Status: in_progress

Problem:
The parser fails to parse function definitions that omit the return type,
which is valid C89/K&R C where the return type defaults to int.

Examples:
  main() { return 0; }           // should be treated as int main()
  Proc_1(x) int x; { ... }      // K&R style, return type defaults to int

Impact:
- Blocks dhrystone benchmark (classic K&R C code)
- Any project using pre-ANSI C function definition style

Fix location:
- src/frontend/parser/declarations.rs: parse_external_decl()
- When parse_type_specifier() returns None, check for identifier followed by '('
  pattern and default to TypeSpecifier::Int
