Task: Fix x86 va_start overflow_arg_area offset for long double params
Status: in_progress

Problem: emit_va_start() computes overflow_arg_area = rbp + 16 + stack_int*8 + stack_fp*8
+ num_named_stack_bytes. But this doesn't account for alignment padding inserted by the
caller when a long double (16-byte aligned) follows 8-byte-aligned stack args.

Example: test_7gp_ld(int a..g, long double h, ...) has 7 GP args (1 on stack = 8 bytes)
plus long double (16 bytes). The caller inserts 8 bytes of padding before the long double
for 16-byte alignment, so actual stack-passed named param space = 8+8+16 = 32.
CCC computes offset = 16+8+16 = 40 instead of correct 48.

Fix: Mirror the caller's alignment padding in calculate_stack_space() by tracking
forward stack offset and adding 16-byte alignment padding before F128Stack/I128Stack args.
