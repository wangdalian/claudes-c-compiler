Fix __attribute__((aligned(expr))) and _Alignas(expr) constant expression evaluation

Bug: The parser only accepts bare integer literals for alignment values in
__attribute__((aligned(...))) and _Alignas(...). Parenthesized expressions
like (64), (1 << 6), and (32 * 2) are silently ignored, resulting in the
default natural alignment being used instead.

Affected locations:
- parser.rs: parse_gcc_attributes() - aligned/__aligned__ attribute
- types.rs: parse_type_specifier() - _Alignas
- types.rs: consume_struct_field_qualifiers() - _Alignas
- declarations.rs: consume_post_type_qualifiers() - _Alignas

Fix: Replace the "check if next token is IntLiteral" pattern with proper
constant expression parsing using the existing parse_expression() +
const_eval infrastructure.

This bug affects all three backends since it's a frontend/parser issue.
Test: /clean_tests/aligned-attr-parenthesized-expr fails on ARM and RISC-V.

Status: IN PROGRESS
